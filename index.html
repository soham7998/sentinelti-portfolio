What is SentinelTI?

SentinelTI is an asynchronous threat intelligence platform that addresses three core failure modes of traditional blacklist-based systems: high false-positive rates, opaque decision-making, and delayed threat visibility.

The framework ingests IP indicators from open-source feeds in real time, enriches them asynchronously with contextual data, and runs a stacking ensemble of Random Forest + XGBoost to produce a dynamic maliciousness risk score. Every score is fully explained using SHAP (SHapley Additive exPlanations), giving SOC analysts transparent, evidence-based reasoning.

Getting Started

Get SentinelTI running in your SOC environment in under 10 minutes.

ℹ Prerequisites

You need Python 3.10+, Docker, a running MongoDB instance, and API keys for VirusTotal and AbuseIPDB.

Installation

Clone the repository and install dependencies:

bash
# Clone the repository
git clone https://github.com/soham7998/sentinel_TI.git
cd sentinel_TI

# Create a virtual environment
python -m venv .venv
source .venv/bin/activate  # Windows: .venv\Scripts\activate

# Install dependencies
pip install -r requirements.txt

Configuration

Create a .env file at the project root with your credentials:

.env
# MongoDB connection
MONGO_URI=mongodb://localhost:27017
MONGO_DB=sentinelti

# Threat intelligence API keys
VIRUSTOTAL_API_KEY=your_virustotal_key_here
ABUSEIPDB_API_KEY=your_abuseipdb_key_here

# Feed configuration
ABUSEIP_LIMIT=200
INGESTION_INTERVAL_SEC=300

# Dashboard
STREAMLIT_PORT=8501

Running with Docker

The recommended production deployment uses Docker Compose:

bash
# Build and start all services
docker-compose up --build -d

# View logs
docker-compose logs -f sentinel

# Access the dashboard
open http://localhost:8501

SentinelTI / Architecture

Architecture Pipeline

A modular, asynchronous pipeline separating fast IOC ingestion from background enrichment — achieving <10s dashboard latency without sacrificing analytical depth.

Full Pipeline

External Threat Feeds

Abuse.ch (primary) · CINS Army (supplementary) — real-time, public IP IOC streams

Fast IOC Ingestion

Normalize, deduplicate, validate. Limit: 200 IOCs per cycle. Stored to MongoDB immediately for zero-latency dashboard visibility.

Async Enrichment

VirusTotal (vendor detections) · AbuseIPDB (historical abuse + confidence) · GeoIP · ASN — runs in background, does not block ingestion

MongoDB IOC Store

Stores timestamps, enrichment status, raw features, and computed scores. Provides both ingestion-time and enrichment-time snapshots.

ML Risk Scoring Engine

Random Forest + XGBoost stacking ensemble. Outputs a continuous risk score in [0, 1] per IP.

SHAP Explainability

Global feature importance + local per-IP SHAP values. Makes every risk score fully auditable.

Actionable SOC Dashboard

Three-tier risk display: LOW / MEDIUM / HIGH. Colour-coded, sortable, evidence-based decisions.

Async Enrichment Detail

The key architectural decision in SentinelTI is the separation of ingestion from enrichment. When a new IOC arrives:

  1. It is immediately stored in MongoDB with a enriched: false flag and displayed on the SOC dashboard as a pending indicator.
  2. A background worker picks it up and calls VirusTotal, AbuseIPDB, GeoIP, and ASN APIs asynchronously.
  3. Once enrichment completes, the feature vector is built and passed through the ML pipeline.
  4. The risk score and SHAP values are written back to MongoDB, updating the dashboard in real time.
Design Rationale

This two-phase approach ensures analysts see IOCs within seconds of ingestion (before enrichment), while full ML-scored records with SHAP explanations appear within ~60 seconds — far below the hours-long lag of traditional blacklist-based systems.

SentinelTI / ML Scoring

Machine Learning Scoring

A stacking ensemble of Random Forest and XGBoost, combined via a linear meta-learner to produce calibrated maliciousness probabilities.

Feature Engineering

Each IP address is represented as a feature vector xᵢ = (f₁, f₂, … f_d) built from enrichment data:

Abuse Reports
📍 AbuseIPDB
Frequency of abuse reports logged against this IP
Confidence Score
📍 AbuseIPDB
Weighted confidence (0–100) of maliciousness
Recency (hours)
📍 Last Report Timestamp
Hours elapsed since last observed abuse activity
VirusTotal Detections
📍 VirusTotal API
Count of vendor detections (e.g., 18/70)
Geo-IP Risk
📍 GeoIP Database
Risk score derived from geographic location
Multi-Source Agreement
📍 Feed Consensus
Number of independent feeds reporting this IP
Attack Type Diversity
📍 AbuseIPDB Categories
Count of distinct attack categories observed
Freshness Feature
📍 Temporal Weighting
Recent threats weighted higher than stale reports

The target variable is binary: yᵢ ∈ {0, 1}, where 1 = malicious, 0 = benign.

Random Forest

RF constructs K decision trees via bootstrap aggregation and random feature sampling. The ensemble malicious probability is:

P_RF(y=1 | xᵢ) = (1/K) · Σ hₖ(xᵢ) for k = 1..K

Classification is applied at threshold τ: if the probability exceeds τ, the IP is flagged malicious.

python
from sklearn.ensemble import RandomForestClassifier

rf = RandomForestClassifier(
    n_estimators=200,
    max_depth=15,
    min_samples_split=5,
    class_weight='balanced',
    random_state=42
)
rf.fit(X_train, y_train)
rf_probs = rf.predict_proba(X_test)[:, 1]

XGBoost

XGBoost builds an additive ensemble of M trees. The malicious probability uses the sigmoid function over the raw additive score:

P_XGB(y=1 | xᵢ) = sigmoid(Σ fₘ(xᵢ)) for m = 1..M
python
from xgboost import XGBClassifier

xgb = XGBClassifier(
    n_estimators=300,
    max_depth=6,
    learning_rate=0.05,
    scale_pos_weight=3,       # handle class imbalance
    use_label_encoder=False,
    eval_metric='logloss'
)
xgb.fit(X_train, y_train)
xgb_probs = xgb.predict_proba(X_test)[:, 1]

Stacking Ensemble

The base learners' probability outputs are combined by a linear meta-learner (logistic regression) trained on a held-out validation fold:

ŷᵢ_stack = β₀ + β₁·P_RF(y=1|xᵢ) + β₂·P_XGB(y=1|xᵢ)

The final malicious probability is then:

P_final(y=1|xᵢ) = sigmoid(ŷᵢ_stack)
Why a linear meta-learner?

The β₁ and β₂ coefficients are directly interpretable — they expose the relative trust placed in each base model. This keeps the entire stack explainable at every hierarchy level while still gaining the performance benefit of ensembling.

python
from sklearn.linear_model import LogisticRegression
import numpy as np

# Stack base predictions as meta-features
Z_train = np.column_stack([rf_val_probs, xgb_val_probs])
Z_test  = np.column_stack([rf_probs, xgb_probs])

meta = LogisticRegression()
meta.fit(Z_train, y_val)

final_scores = meta.predict_proba(Z_test)[:, 1]
print(f"RF weight: {meta.coef_[0][0]:.3f}, XGB weight: {meta.coef_[0][1]:.3f}")

SHAP Explainability

SHAP (SHapley Additive exPlanations) provides both global model-wide importance and local per-IP attribution — making every risk score fully auditable by SOC analysts.

Global Feature Importance

Global SHAP aggregates feature contributions across all IP instances to reveal the dominant threat drivers:

Iⱼ = (1/n) · Σ |φᵢⱼ| for i = 1..n

Local Per-IP Explanation

For each IP, SHAP decomposes the final risk score into per-feature contributions:

P_final(y=1|xᵢ) = φ₀ + Σ φᵢⱼ for j = 1..d

A positive φ value pushes the score toward malicious; a negative φ reduces it.

python
import shap

# Global explainer on stacking meta-learner
explainer = shap.TreeExplainer(xgb_model)
shap_values = explainer.shap_values(X_test)

# Global summary plot
shap.summary_plot(shap_values, X_test, feature_names=feature_cols)

# Local explanation for a single IP
ip_idx = 0   # index into X_test
shap.force_plot(
    explainer.expected_value,
    shap_values[ip_idx],
    X_test.iloc[ip_idx],
    feature_names=feature_cols
)

SentinelTI / Risk Scoring

Risk Scoring & Metrics

Three-tier stratification gives SOC analysts clear, actionable priority levels — eliminating the equal-urgency problem of binary blacklists.

Risk Level Classification

LOWScore: 0.0 – 0.3Benign or stale reports. No action needed.
MEDIUMScore: 0.3 – 0.7Suspicious or mixed signals. Review recommended.
HIGHScore: ≥ 0.7Active threat. Immediate action required.

⚠ Threshold Tuning

The default thresholds (0.3, 0.7) are calibrated for the SOC target FPR of <5%. Adjust in config/risk_thresholds.yaml if your environment has different sensitivity requirements.

SentinelTI / API Reference

API Reference

REST endpoints for programmatic access to SentinelTI's ingestion, scoring, and explanation pipeline.

ℹ Base URL

All endpoints are relative to http://localhost:8000/api/v1 (or your deployed host). Authentication is via Bearer token set in the Authorization header.

Endpoints

POST /score

Submit one or more IP addresses for immediate scoring. Returns risk score, level, and SHAP attribution.

Request
Response
Python
POST /api/v1/score
Authorization: Bearer <token>
Content-Type: application/json

{
  "ips": ["137.184.9.29", "45.33.32.156"],
  "explain": true
}
{
  "results": [
    {
